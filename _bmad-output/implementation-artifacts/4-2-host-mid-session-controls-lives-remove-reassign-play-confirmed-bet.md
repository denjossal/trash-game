# Story 4.2: Host mid-session controls ŌĆö Lives, remove, reassign (play-confirmed bet)

Status: done

## Story

As a Host,
I want to adjust Lives, remove a departed Player, and hand off the Host role,
so that I can keep the night moving without restarting the Table ŌĆö all from the ŌÜÖ Host Controls sheet, never on a Player's turn screen.

## Context & Why This Story Matters

This is the **second and last** story of Epic 4 ("Conduct the night") and the **first full-stack** story of the epic. Story 4.1 shipped the **shell**: the Host-only `ConductorBar` (Deal / Showdown / Re-deal primaries) and a one-level `HostControls` modal sheet that opens from the ŌÜÖ affordance ŌĆö but the sheet **body is an empty placeholder** ("More controls coming soon"). This story **fills that sheet** with the three FR-14 controls and wires the two missing server handlers.

The wire contract already names all three intents (`shared/src/types.ts:173-175`), and **`hostSetLives` already has a complete server handler** (`handlers.ts:275-311`, lobby-only) **and a complete client send seam** (`sendHostSetLives` / `buildHostSetLivesIntent`). What is **missing**:

- **Server:** `hostSetLives` is gated `phase === "lobby"` only ŌĆö this story makes it work **mid-session** (between Rounds and beyond, per the M1 decision below). And there is **no handler at all** for `hostRemovePlayer` / `hostReassign` (the `dispatch.ts:175-177` default throws `phase-illegal` for both).
- **Client:** the Lives stepper lives **inline on Lobby** today (`Lobby.svelte:59-75`) ŌĆö this story **moves it into the ŌÜÖ sheet** (user decision; DESIGN.md:186). There are **no builders or send seams** for `hostRemovePlayer` / `hostReassign`.

**This is a FULL-STACK story** (server handlers + client UI), but **NO `shared/src/types.ts` change** ŌĆö all three intents are already in the `Intent` union, and all error reasons are already frozen (reuse `not-host` / `stale-phase` / `phase-illegal`).

### Resolved product decisions (settled at create-story, baked into ACs)

1. **M1 ŌĆö clamp-vs-top-up (owed since Epic 3 retro; the one open item with a correctness consequence).** **DECISION: "set ongoing, never revive."** A mid-session `hostSetLives(n)` sets `startingLives = n` AND sets `lives = n` for **every still-alive Player** (`isAlive === true`). An **eliminated (0-life, `isAlive === false`) Player is NEVER touched and NEVER un-eliminated** ŌĆö permanent elimination (Architecture D1, Story 3.5) is preserved. A raise tops up live Players; a lower clamps them; the dead stay dead. *(Mirrors the existing lobby `handleHostSetLives` lives-sync at `handlers.ts:308-309`, but guarded to alive seats only.)* `[Source: implementation-readiness-report:225-226/260; epic-3-retro:83-84; prd FR-14:258.]`

2. **Lives stepper home ŌĆö ŌÜÖ sheet only.** The inline Lobby stepper is **removed**; the ŌÜÖ Host Controls sheet becomes the **sole** home for the Lives stepper (alongside remove + reassign). *(DESIGN.md:186 ŌĆö "the controls overlay contains the Lives stepper.")* Lobby keeps the roster + pips only.

3. **Phase gate for remove/reassign ŌĆö any phase, server handles the mid-Round edge.** Both are accepted at any phase the Host can reach the sheet from. Remove normally excludes the Player from the next Deal; mid-Round (`phase === "turns"`) removal of the **current-turn** Player advances `currentTurnId = nextAliveSeat(removed)` and adds the removed Player to `acted` so the one-pass still satisfies the `turns ŌåÆ allActed` condition. *(epics.md:851; AR-5; architecture.md:339-344.)*

## Acceptance Criteria

1. **AC-4.2.1 ŌĆö Mid-session Lives change works end-to-end ("set ongoing, never revive").** **Given** the Host on the ŌÜÖ Host Controls sheet **When** the Host changes Lives via the stepper (clamped 1ŌĆō5) **Then** the client sends `hostSetLives{phaseToken, lives}` via the existing `sendHostSetLives` seam, the server accepts it **at any phase the sheet is reachable from** (NOT lobby-only anymore), sets `startingLives = clamped` AND sets `lives = clamped` for **every Player with `isAlive === true`**, leaves every **eliminated** (`isAlive === false`) Player **untouched** (never un-eliminated), persists, and fans out the new value to every device. *(FR-14, AR-6; M1 decision; epics.md:845-847.)*

2. **AC-4.2.2 ŌĆö Lives change carries + tolerates the phaseToken (stale rejected).** **Given** a mid-session `hostSetLives` **Then** the intent carries the current `state.phaseToken`; a stale/double-tapped copy is rejected with `stale-phase` (silently swallowed client-side). The handler **does NOT bump** `phaseToken` (a config change is not a phase transition ŌĆö same as the lobby behavior; the token is *accepted-but-not-guarded* in lobby where it is 0, and *guarded-but-not-bumped* mid-session). *(AR-6; epics.md:847; handlers.ts:275-311 precedent.)*

3. **AC-4.2.3 ŌĆö Remove a Player (error-tinted affordance, with confirm).** **Given** the roster in the ŌÜÖ sheet **When** the Host taps a Player's **error-tinted** remove affordance **and confirms** **Then** the client sends `hostRemovePlayer{phaseToken, playerId}` via a NEW send seam; the server (NEW `handleHostRemovePlayer`) removes that Player from the roster (`players` array), and the removed Player is **excluded from the next Deal**. The remove affordance requires an explicit confirm step (no accidental single-tap removal). *(FR-14, AR-5; epics.md:849-851; DESIGN.md:186.)*

4. **AC-4.2.4 ŌĆö Mid-Round removal of the current-turn Player advances the turn (no stall).** **Given** `phase === "turns"` and the Host removes the Player who is `round.currentTurnId` **Then** the server advances `round.currentTurnId = nextAliveSeat(removedSeatIndex)` AND adds the removed Player's id to `round.acted` so the one-pass can still satisfy `turns ŌåÆ allActed` (the table does NOT stall on a gone Player's turn). **Given** the removed Player is NOT the current-turn Player **Then** the current Round is **not rewritten** ŌĆö removal resolves at the next Showdown/Re-deal (the removed seat simply gets no card on the next `dealRound`, which deals to `isAlive` seats only). *(AR-5; epics.md:851; architecture.md:339-344; engine.ts:88-109 `nextAliveSeat`.)*

5. **AC-4.2.5 ŌĆö Reassign Host (hand off the conductor role).** **Given** the "Make someone else host" picker in the ŌÜÖ sheet **When** the Host reassigns to another Player **Then** the client sends `hostReassign{phaseToken, playerId}` via a NEW send seam; the server (NEW `handleHostReassign`) sets `table.hostId = playerId`, the **new Host gains the conductor controls** (the bar + sheet now render for them on their next projection), the **former Host becomes a regular Player**, and **exactly one Host exists at any time**. An **eliminated Host keeps conducting** until they reassign or disconnect (the picker is available even when the Host is eliminated ŌĆö the bar already mounts on `eliminated` for a Host, Story 4.1 change-log). *(FR-14, AR-5; epics.md:853-855; architecture.md:335-338.)*

6. **AC-4.2.6 ŌĆö All three controls are Host-only and never on Your Turn.** **Given** any of the three controls **Then** they live ONLY inside the ŌÜÖ Host Controls sheet, which opens ONLY from the Host-only `ConductorBar` (already gated `isHost` + absent on `yourTurn`, Story 4.1). A non-Host never reaches them; they never appear on any Player's Your Turn surface. *(FR-14, NFR-9; epics.md:857-859.)*

7. **AC-4.2.7 ŌĆö seatIndex collision fixed (regression guard for removal).** **Given** a Player has been removed and a new Player joins **Then** the new Player gets a seatIndex that **does not collide** with any existing seat. The current `seatIndex = host.table.players.length` (handlers.ts:227) is replaced with a monotonic derivation `max(existing seatIndex) + 1` (or 0 for an empty roster) so a post-removal join never reuses a live seat's index. *(deferred-work #44; architecture.md:316-320 "seatIndex immutable-for-life, never re-indexed/reused"; engine.ts:88-109 `nextAliveSeat` walks by seatIndex.)*

8. **AC-4.2.8 ŌĆö Eyes-Up gate: the sheet holds ONLY the three controls.** **Given** the ŌÜÖ sheet with the three controls built **Then** it contains ONLY {Lives stepper, roster-with-remove, reassign picker} + the heading + close affordance ŌĆö NO turn timer, NO activity/event log, NO player-status dashboard, NO ambient/idle content, nothing to scroll beyond the controls themselves. *(Mary review; Eyes-Up standing gate G1 / NFR-9; epics.md:833-835; epic-3-retro:98.)*

### The guard pattern these handlers must follow (the chokepoint)
The canonical accepted-path order (from `requirePhaseConductor` at `handlers.ts:684-719` and `handleDeal` at `:339`): **shape ŌåÆ table-null ŌåÆ not-host ŌåÆ checkPhaseToken ŌåÆ [phase/state guard] ŌåÆ mutate ŌåÆ bump ŌåÆ persist.** For the two new handlers, do NOT call `requirePhaseConductor` (it hard-codes a single `expectedPhase` gate, and remove/reassign are accepted at ANY phase per the decision) ŌĆö instead inline the same checks in the same order WITHOUT a phase gate. Throw `IntentError` (caught by `dispatch.ts:179-185` ŌåÆ `pushError`, NO fan-out on error). On success, `dispatch` calls `fanOut` after the handler returns.

`hostSetLives` is special: it does NOT bump `phaseToken` (config change, not a phase transition ŌĆö `handlers.ts:308-309` precedent). The two new handlers DO `bumpPhaseToken` (a roster/host mutation is a state change; bumping protects against double-tap and keeps the token monotonic).

### The mid-Round removal sequence (AC-4.2.4 ŌĆö get the ORDER right)
`nextAliveSeat(players, fromSeatIndex)` (engine.ts:88-109) **asserts** `fromSeatIndex` exists in the passed array (throws if `startPos === -1`, engine.ts:95-97). So when removing the current-turn Player at `phase:"turns"`:
1. Compute the advance FIRST, while the target seat still exists: `const nextId = nextAliveSeat(host.table.players, target.seatIndex);` ŌĆö `nextAliveSeat` walks right by seatIndex and returns the next `isAlive` seat's id (skipping the target if it were still alive; here the target IS still in the array and alive, but we're walking PAST it, so it returns the seat after).
2. Then remove: `host.table.players = host.table.players.filter((p) => p.id !== target.id);`
3. Set `host.table.round.currentTurnId = nextId;` and `host.table.round.acted.push(target.id);` ŌĆö the gone Player counts as "acted" so `allAlivePlayersActed(round, players)` (engine.ts:227-230, checks `!p.isAlive || acted.has(p.id)`) can still reach true. NOTE: since the target is removed from `players`, `allAlivePlayersActed` no longer iterates them anyway ŌĆö but adding to `acted` is harmless and matches the epics.md:851 spec literally; keep it for fidelity + in case any consumer still references the id.
4. `bumpPhaseToken`, persist.

Watch the heads-up edge: if removing the current-turn Player in a 2-player round leaves 1 alive, `nextAliveSeat` over the post-removal array returns the lone survivor (or the start seat fallback, engine.ts:107). The round can't complete into a meaningful showdown with 1 player ŌĆö but the Host removed someone mid-round deliberately; the next reveal/re-deal resolves it (`resolveShowdown` / `handleDeal Ōēź MIN_PLAYERS`). Do NOT add special win logic in the remove handler.

### M1 ŌĆö "set ongoing, never revive" (the resolved decision; AC-4.2.1)
The CURRENT lobby loop syncs ALL seats: `for (const p of host.table.players) p.lives = next;` (handlers.ts:309). Mid-session this would **revive an eliminated player** (set a 0-life `isAlive:false` seat back to `next` lives). **The fix:** gate on `isAlive` ŌĆö `for (const p of host.table.players) if (p.isAlive) p.lives = next;`. This is safe in lobby too (pre-Deal every seat is `isAlive:true`), so the SAME gated loop works for both lobby and mid-session ŌĆö you can apply the `if (p.isAlive)` gate unconditionally. `startingLives` is still set for all (it's the table config, used by `handleNewGame` / `handleJoinRoom` for future seats).

### seatIndex collision (AC-4.2.7 ŌĆö deferred-work #44, the regression trap)
`handleJoinRoom` assigns `seatIndex: host.table.players.length` (handlers.ts:227). This is safe ONLY while the roster is append-only. Once `hostRemovePlayer` splices the array, a roster of `[seat0, seat2]` (seat1 removed) has `length === 2`, so the next join gets `seatIndex: 2` ŌĆö **colliding with the existing seat2**. `nextAliveSeat` walks by seatIndex and would then find two seats with index 2 (its `sort` + `findIndex` would mis-walk). **Fix:** derive `max(seatIndex)+1`. Architecture.md:316-320 mandates "seatIndex immutable-for-life, never re-indexed **or reused**" ŌĆö `max+1` honors "never reused." No persistence change (derives from persisted `players[].seatIndex`).
