## Epic Summary

Once the Last Player has acted, the Host triggers the simultaneous reveal; the app flips every Card at once, computes the lowest value, highlights the Loser(s) (ties and all-tied included), deducts a Life, eliminates anyone at zero, declares a winner when one remains, and re-deals in one tap with the Loser starting the next Round. Delivers the reveal beat (SM-3) and **closes the core game loop end-to-end** — deal → turn → reveal → resolve → re-deal → winner → "one more."

## What Went Well (lock in as deliberate practice)

- **Contract-up-front held for a THIRD epic running.** Zero `types.ts` changes across all six stories. 3.6's `newGame` extended an Intent member that Story 1.3 had already named (alongside `dealAgain`), and the `gameOver`→`lobby` phase edge was anticipated in the contract — so the game-lifecycle closer introduced no late intent. This is the same Epic-1 decision #2 dividend, now proven durable.

- **The Epic-2-retro gate (Action 3, Playwright harness) paid for itself on its first run.** Standing the Chromium harness against live `wrangler dev` BEFORE Epic 3 immediately caught a real production defect that all 222 unit/DO tests missed: `createSocket` omitted `party: "table"`, so partysocket defaulted to `/parties/main/<code>` — a path the server REFUSES — meaning **every real browser connection was being refused**. The unit/DO suite bypasses HTTP routing, so it could never see this. Exactly the integration gap the retro predicted manual ACs were hiding. (commit `ac4c8fe` / fix `c5f78d2`.)

- **Action 4 (engine-primitive assert-vs-trust) landed cleanly — and the review found the gap we'd half-missed.** 3.1 folded the "assert-in-primitive" hardening into `dealRound` (deck-coverage + alive-starter) and `nextAliveSeat` (unknown-seat) as planned. The code review then caught that `resolveShowdown` itself had been exempted from its own policy: an unknown `previousStartingPlayerId` → `bySeat[-1]` TypeError, and empty contenders → `Math.min()===Infinity`. **Both failing inputs were precisely the eliminated-previous-host case that 3.4's `dealAgain` introduces** — caught one story before the caller arrived.

- **Clone-the-chokepoint discipline kept reviews clean.** Every new handler copied the `guard → mutate → bump → persist` shape `handleDeal` set in Epic 2. `handleReveal` cloned `handleDeal`; `handleDealAgain` cloned the deal chokepoint; `handleNewGame` cloned `handleDealAgain` through the shared `requirePhaseConductor` helper (its 4th caller). The result: **3.5 and 3.6 shipped with zero review findings.**

- **Forward-built branches accelerated delivery.** Story 1.4's pre-built `revealed` projection branch meant 3.2 only had to make `revealed` REACH `true` — the genuine RED was the wire path, not the projection. Likewise 3.3 built the loser-highlight as a dormant consumer keyed on `loserIds`, lit up automatically when 3.4 wired the producer.

- **RED-first stayed unbroken across all six stories** (3.1 17-fail→pass, 3.2 wire-path RED, 3.3 6/8, 3.4 multi-front, 3.5 aria-live RED, 3.6 three-front). No story shipped without a confirmed failing test first.

## What Was Hard (recurring patterns)

- **The two qualitative gates were marked `done` in code without the live observation actually being run (the #1 theme — and it's the SAME pattern Epic 2 flagged).** AC-3.3.7 (pre-mortem A: squirm-beat and reveal-beat tuned TOGETHER in one live session, freeze criterion "the pair lands; ship") and AC-3.5.7 (Decision #6: a real eliminated player — 9-year-old / non-gamer — observed to stay engaged) were both honestly recorded as non-code time-boxed gates, but **neither was performed.** Epic 2's retro warned "qualitative ACs need a re-runnable home or they pile up"; Epic 3 added two more on top of the three Epic-2 manual ACs. The harness discharges the *deterministic* ones (peek auto-hide, unaided turn, multi-device) but **cannot discharge a "does it feel right / does a kid stay engaged" gate** — those genuinely need a human at a table.

- **The `showdown` phase literal is now transient/internal — a readable-only-if-you-know-it subtlety.** 3.4's resolve-at-reveal decision means the wire NEVER rests at `showdown`; resolution runs synchronously inside `handleReveal` and lands at `roundResult`|`gameOver`. The router still sends a `revealed===true roundResult` to the Showdown *surface* by design (route-from-state.ts:53 before :48). This is correct but counter-intuitive — it's documented in the handler SCOPE comment specifically so a future reader doesn't "fix" the router by reordering it. A latent trap for Epic 4/5 maintainers.

- **`@cloudflare/vitest-pool-workers` cold-run flakiness is now a documented recurring pattern.** A `.do.test.ts` flaked once on a cold full-suite run in 3.5, passed on re-run; 3.6 pre-noted the re-run protocol as a guard and it did not recur. Non-blocking, but it's real friction that every server-touching story now carries a "re-run before treating as regression" footnote.

- **Co-winner grammar imperfection knowingly accepted.** The shared-win case joins all names into the single `winner(name)` slot ("…and Ben wins it") to preserve one source string (UX-DR16). A deliberate trade of grammatical polish for copy-source simplicity — flagged so it's a choice, not a bug, if it ever grates in play.

## Key Lessons

1. **A harness can only discharge deterministic ACs — "does it feel right" still needs a human at a table.** The Playwright gate was a triumph for the mechanical manual ACs, but AC-3.3.7 (the pair lands) and AC-3.5.7 (the kid stays engaged) are irreducibly qualitative. They don't need a *harness*; they need a *scheduled play session*. Conflating the two is how they keep slipping.
2. **Pre-built / forward-deferred code comes due at its first real caller — and the review is where the half-missed edge surfaces.** 3.1's `resolveShowdown` precondition gap, 3.2's `undefined`-hand projection, and Action-4 all converged on 3.4's `dealAgain`. The pattern from Epic 2's lesson 4 repeated exactly — and the code review (not the build) is what caught the resolver exemption.
3. **Clone-the-chokepoint + contract-up-front is now a three-epic-proven engine for clean reviews.** Two stories this epic shipped with zero findings. The structural discipline (one persist chokepoint shape, one frozen contract) is *why*.
4. **Document the counter-intuitive invariant inline, at the trap.** The transient `showdown` literal and the router precedence are correct-but-surprising; the SCOPE comment that explains them is what stops a future maintainer from "fixing" working code. Make this a habit, not a one-off.
