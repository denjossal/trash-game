## What Went Well (lock in as deliberate practice)

- **Contract-up-front held for a FOURTH epic running.** Zero `types.ts` changes — and this time on a genuinely full-stack story (4.2 added two new server handlers + new client send seams) that still only EXTENDED Story 1.3's frozen `Intent` union. The Epic-1 decision-#2 dividend is now four epics deep.

- **Clone-the-chokepoint discipline scaled to the new "no phase gate" handlers.** Both `handleHostRemovePlayer` and `handleHostReassign` copied the established `shape → table-null → not-host → checkPhaseToken → mutate → bump → persist` shape, simply dropping the phase gate (host controls are reachable from any phase). The structural template absorbed a new handler *class* without inventing a new shape.

- **The code review caught what the build, typecheck, and existing tests all missed (the headline lock-in — Dennis's chosen priority).** 4.2 shipped "green," but the adversarial review layers found that `handleHostRemovePlayer` advanced `currentTurnId` on a current-turn removal yet **never re-ran `maybeCompletePass`**. Removing the *last un-acted* on-turn Player stranded the Round in `turns`: the Host could re-take a turn, and `revealAll` (gated on `allActed`) was unreachable. **The Blind Hunter and the Edge Case Hunter found it independently.** The review also caught a follow-on hazard — clearing `nextStartingPlayerId` on removal would have crashed the next `dealAgain`'s stashed-starter invariant — so the fix re-seats it via `nextAliveSeat` instead.

- **The live playtest has demonstrated teeth.** 4.1's change log records a playtest-discovered bug: an eliminated Host knocked out mid-game routed to the Eliminated spectator surface but REMAINED Host — yet the conductor bar was excluded from `eliminated`, stranding a 4-player game ("You're out" with no Re-deal). Hands-on clicking found what the unit suite could not. This is the Epic-3-retro thesis ("a harness can't discharge does-it-feel-right") proven in the field.

- **M1 was decided BEFORE the build, not as a mid-build `[ASSUMPTION]`** (Epic-3 Action 4 discharged). "Set ongoing, never revive" was settled at 4.2 create-story and baked into AC-4.2.1; the code gates the lives-sync loop on `isAlive` so an eliminated 0-life seat is never un-eliminated. Permanent elimination (Architecture D1) is preserved by construction.

- **RED-first stayed unbroken under full-stack load.** 4.2's 13 new server tests failed against the lobby-only `hostSetLives` + `default → phase-illegal` baseline before implementation, all green after. 4.1's component tests failed against the missing component / 1.9a stub first.

- **`handleHostRemovePlayer` is now the canonical order-of-operations trap.** Three mutations must happen in a precise order relative to the roster splice: `nextAliveSeat` (turn advance) computed BEFORE the splice (the engine asserts the seat exists), `nextStartingPlayerId` re-seat BEFORE the splice, and `maybeCompletePass` AFTER it (it reads the now-current roster). Reordering any of them reintroduces a stranding or dangling-id bug. Documented inline at the trap — same habit as Epic 3's transient-`showdown` router invariant.

- **`@cloudflare/vitest-pool-workers` cold-run flakiness recurred (and the re-run protocol worked).** A full-server run flaked once during the 4.2 code-review verification; the documented "re-run before treating as a regression" protocol cleared it (210/210 on re-run). ## Key Lessons

1. **For a full-stack, state-mutating handler, the code review — not the build — is where correctness lives.** 4.2 passed build + typecheck + its original 13 tests and still stranded the round. The adversarial review layers (independently, twice over) found it. Make that review depth standard for any handler that mutates `round` / `currentTurnId` / `acted` / phase. *(Dennis's chosen lock-in.)*
2. **Hands-on play finds what no harness can — and it has now paid off twice.** 4.1's eliminated-Host stranding bug surfaced only by clicking a real game. The qualitative gate isn't bureaucratic; it's a distinct bug-finding channel. Stop deferring it.
3. **Decide open product questions before the build, document order-of-operations invariants at the trap.** M1 settled at create-story (no mid-build assumption); the removal handler's compute-before/after-splice ordering documented inline so a future reorder doesn't silently reintroduce the stranding bug.
4. **Clone-the-chokepoint + contract-up-front is now a FOUR-epic-proven engine.** It absorbed an entirely new handler class (no-phase-gate host controls) with zero contract change and a clean structural fit. Reaffirm it as deliberate practice, not habit.

## Next Epic Preview — Epic 5: Scale to the whole table

**Scope:** The app silently scales the Deck to headcount — one deck for ≤10 Players, two merged decks for 11–20 — so the Host never thinks about it, and the Loser computation stays correct and unambiguous at a full 20-Player table where duplicate values (and more frequent ties) are expected and accepted. **FRs:** FR-13 (and hardens FR-10 / FR-11 at scale).

**Dependencies on prior epics — VERIFIED solid this retro (Dennis chose "verify first"):**
- **Resolution is parameterized 2..20 from day one (Decision #7).** Confirmed against the code: `resolveShowdown` has four parameterized `for (n = 2; n <= 20; n++)` tests (`engine.test.ts:598-639` — single-lowest, all-tied, all-tied-to-zero shared win, two-way tie), and `assertDealable` accepts any table size 2..20 (`validate.test.ts:126`). **Epic 5 does NOT reopen the core resolution function.**

**Therefore Epic 5 shrinks to:** the deck-size mapping (1 deck ≤10 / 2 merged decks 11–20) + one boundary integration test (10↔11 player threshold). The hardest correctness work (ties at scale) was already paid down in Epic 3.
