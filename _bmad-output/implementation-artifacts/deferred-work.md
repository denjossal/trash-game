## Standing obligations from: Story 1.4 (SM-6 inference channels)

> SM-6 is "no card SEEN **or INFERRED**." Story 1.4 closed the SEEN channel mechanically (`projectStateFor` + the standing negative-assertion test in `server/src/project-state.test.ts`) and pinned inference channel (a) constant message shape. The two remaining inference channels are NOT code in 1.4 — there is no turn/peek/swap code yet — so they are carried here as STANDING review obligations, each tagged with its owning Epic 2 turn story. (On-disk mapping: peek = Story 2.5, draw = Story 2.6; the epics' Mary-review parenthetical's "2.7" is a typo — Epic 2 has no Story 2.7.)

- **(b) Timing-indistinguishability of turn responses** [Story 2.4 (swap/keep), Story 2.5 (peek), Story 2.6 (last-player swap / drawFromDeck)] — a turn-scoped response (`swap` / `keep` / `peek` / `drawFromDeck`) MUST be timing-indistinguishable by card value: NO value-dependent server-latency branch. A handler that does more work (or less) depending on the hidden card rank would leak the value through response timing even with the value absent from the payload. Each of these stories must keep its handler's work-per-intent independent of any hidden card value.
- **(c) No behavioral tell on any surface** [Story 2.4 (swap/keep), Story 2.5 (peek), Story 2.6 (last-player swap / drawFromDeck)] — no rendered surface may expose a behavioral tell (a glow, an enabled/disabled state, a count, or any other value-dependent UI affordance) that lets a neighbor infer a hidden card. The value-free server-computed fields (`isLastPlayer`, `isAlive`, etc.) exist precisely so the client never derives turn/elimination/value logic locally; these stories must not introduce a client-side affordance keyed on a hidden card value.
